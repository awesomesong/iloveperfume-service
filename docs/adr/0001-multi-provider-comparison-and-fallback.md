# ADR-0001: 멀티 LLM 프로바이더 비교 + 첫 토큰 전 폴백

## Status

Accepted — 2026-07-16

## Context

체크리스트 7번(POST /messages 라우트) 작업 중, 대화하면서 두 가지 요구가 나왔다: (1) 어떤 모델이 추천 품질(클릭 전환율)이 더 좋은지 비교하고 싶다, (2) 한 프로바이더가 장애일 때 서비스가 죽지 않았으면 한다.

PLAN.md 4번 섹션은 "LLM 응답 품질 평가(golden dataset)"를 가설 검증 후로 미뤄뒀지만, 이건 별도 채점 인프라를 새로 만드는 얘기였다. 지금 결정은 채점 인프라 없이, 이미 있는 지표(ClickEvent 기반 클릭 전환율)를 프로바이더별로 나눠서 보는 것뿐이라 그 결정과 충돌하지 않는다.

ADR-0000은 "지금 안 써도 되는 복원력은 미리 만들지 않는다"고 했다. 완전 자동 장애 대응(스트리밍 도중 실패까지 무중단 전환)은 여전히 과할 수 있어서, 폴백 범위를 첫 토큰 전송 전으로 제한해 복잡도를 낮췄다.

## Decision

- Conversation 생성 시 Claude/GPT/Gemini 중 하나를 무작위로 배정하고(`assignedProvider`), 대화 내내 유지한다 — 같은 대화 안에서 모델이 바뀌면 비교 지표가 왜곡되기 때문.
- 폴백은 **첫 토큰을 클라이언트로 보내기 전까지만** 허용한다. 배정된 프로바이더 호출이 스트리밍 시작 전에 실패하면 다음 프로바이더로 자동 전환한다(사용자는 못 알아챔). 이미 스트리밍이 시작된 뒤 실패하면 그대로 에러 처리한다 — SSE 실시간 스트리밍 UX(PLAN.md 확정 사항)를 지키기 위해 전체 응답을 서버에서 버퍼링하는 방식은 채택하지 않는다.
- Recommendation에는 실제로 응답을 생성한 프로바이더/모델(`modelProvider`, `modelId`)을 기록한다 — 폴백이 발생하면 배정된 프로바이더와 다를 수 있어서 비교 지표의 정확도를 위해 필수.

## Consequences

- API 키 3개(ANTHROPIC_API_KEY, OPENAI_API_KEY, GOOGLE_GENERATIVE_AI_API_KEY)를 운영해야 하고, 토큰 비용이 프로바이더 3곳에 분산된다.
- 클릭 전환율을 프로바이더별로 나눠 보려면 표본이 3배로 필요하다 — 가설 검증 기간이 길어질 수 있다.
- 스트리밍 시작 후 실패는 폴백 없이 에러로 끝난다. 이 비율이 실제로 문제될 만큼 높아지면(신호가 오면) 그때 버퍼링 방식이나 부분 재시작을 재검토한다.
