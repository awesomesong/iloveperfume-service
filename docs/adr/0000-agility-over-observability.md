# ADR-0000: 민첩성을 장애 자동 대응보다 우선한다

## Status

Accepted — 2026-07-15

## Context

I Love Perfume 리뉴얼은 아직 검증되지 않은 가설("AI 챗봇이 향수를 추천하면 사용자가 구매 링크를 클릭하는가") 하나를 확인하기 위한 MVP다. 개발자는 1인이고, 상시 on-call 체계는 없다.

아키텍처 특성은 서로 상충하는 경우가 잦다 — 예를 들어 장애를 자동으로 탐지하고 대응하는 체계를 갖추려면 그만큼 인프라와 코드에 관측·복구 로직을 미리 넣어야 하고, 이는 가설이 틀렸을 때 버려야 할 코드가 늘어난다는 뜻이다. 반대로 민첩성을 우선하면 장애 대응이 늦어질 수 있다.

## Decision

아키텍처 특성의 우선순위를 다음과 같이 정한다.

1. **민첩성** — 가설이 틀렸다는 게 확인되면 방향을 빠르게 바꿀 수 있어야 한다. 지금 안 써도 되는 확장성·복원력은 미리 만들지 않는다.
2. **수분 내 장애 탐지·자동 대응** — 완전 자동화는 아니어도, 문제가 생겼을 때 사람이 몇 분 안에 알아챌 수 있는 최소한의 관측(Sentry, Uptime 체크)은 갖춘다.

두 특성이 충돌하는 결정에서는 애매하면 더 민첩한 쪽을 선택한다.

## Consequences

- 인프라·코드 구조를 결정할 때 "지금 검증에 필요한가"를 1차 기준으로 삼는다. 예: apps/realtime을 지금 만들지 않는 것(저장소 구조 결정), 테스트 스위트를 최소로 유지하는 것(PLAN.md 4번 섹션)은 모두 이 ADR에서 파생된 결정이다.
- 24/7 자동 복구, 다중 리전 이중화 같은 고가용성 설계는 하지 않는다. 장애가 나면 사람이 수분 내 알아채고 수동으로 대응하는 것을 목표로 한다.
- 가설이 검증되어 실제 트래픽이 늘면 이 우선순위 자체를 재검토해야 한다 — PLAN.md 8번 "스케일 전환점"에 그 트리거 조건을 기록해둔다.
